1. What is Domain-Driven Design

A importância do domínio

Nunca será possível criar softwares complexos sem bom conhecimento do domínio.
O domínio é sempre muito conhecido pelos seus especialistas: no banco, por exemplo, o bancário.
Tudo deve sempre começar pelo domínio.
O propósito do software sempre vai ser melhorar um domínio.

Como o software pode ser harmonioso com o domínio?

O software deve espelhar o domínio.

Como fica a aparência de um software orientado ao domínio?

Alguém que não tem conhecimento de domínio deve ser capaz de aprender um monte sobre o domínio apenas lendo o código dos modelos de domínio.

Modelo de domínio

É a abstração de um domínio. É o conhecimento presente na cabeça de um especialista (o bancário, por exemplo) expresso de alguma forma acessível.
Um diagrama não é um modelo de domínio. O diagrama é uma forma de expressar a ideia do que é um modelo de domínio, assim como isso pode ser expresso em frases, ou em código. O modelo de domínio será o conjunto de regras fortemente definidas que definem esse conhecimento.
É essencial para o processo de design de sistemas e não é possível prosseguir sem ele, já que o mundo ao nosso redor é demais para que um cérebro possa armazenar todas essas informações.
Os modelos precisam sair da cabeça e serem expressos de forma a mostrá-los aos especialistas de domínio, designers e desenvolvedores → o modelo precisa ser comunicável. Seja por um diagrama, por uma linguagem, por texto.
O design de código só pode ser iniciado após o modelo de domínio ser definido.

Design de código vs. design de software

Pense como o projeto de uma casa. O design de software é o projeto arquitetônico: as paredes, as vigas, o projeto elétrico e hidráulico e a disposição dos cômodos; já o design de código é a organização dos interiores: qual a cor das paredes, a posição dos quadros, quais quadros devem ser comprados, os móveis.
Design de software tem muito mais impacto e é muito mais difícil de ser revertido após estabelecido — uma nova “obra” terá que ser feita.
Design de código é mais simples, basta comprar outros móveis, reorganizar a casa, fazer uma nova pintura, sem impactos no software.
Sem o design de código, porém, o produto final nunca será de qualidade

2. The Ubiquitous Language

Necessidades de uma linguagem padrão

Comunicação de desenvolvedores não é a mesma usada pelos especialistas de domínio.
Aqueles que dominam de negócio com certeza utilizam um jargão próprio.
Enquanto isso, desenvolvedores tendem a incluir conceitos técnicos (herança, polimorfismo, etc.).

Linguagem ubíqua

Linguagem padronizada baseada no modelo de domínio.
Altamente enfatizado (princípio base do DDD) que todas as comunicações devem ser realizadas por meio da linguagem ubíqua.
Ambas as partes (desenvolvedores e especialistas de domínio) participam da criação dessa linguagem
Por exemplo, no caso da Havan, o pré-pedido comprado por pack e por peça poderia gerar uma confusão grande. Essa é a linguagem usada pela área de negócio, mas ficou acordado que “pré-pedido por variação” seria a nomenclatura correta e evitaria essa ambiguidade para ambas as partes.
No momento de criação da linguagem ubíqua, todos devem estar focados em tal objetivo usando tal linguagem já definida sempre que necessário.

Por que desenvolvedores já devem iniciar a implementação dos principais conceitos do modelo no código?

Dessa maneira, a codificação do domínio começa a já mapear o modelo de domínio e a linguagem ubíqua na forma de código antes de qualquer implementação. Isso deixa o código mais legível e se paga mais tarde no projeto, quando más decisões nesse momento teriam que ser refeitas por questões de código.

Algumas formas de expressar modelos de domínio em documentos

UML (Unified Modeling Language)

Uma boa opção, mas geralmente de entendimento restrito aos desenvolvedores.
Funciona bem para modelos pequenos, mas modelos grandes podem se tornar uma dor de cabeça.

Documentos escritos

Cada documento apresenta parte do modelo de forma isolada, mesmo que eles façam parte de um todo maior.
Passagens de texto muito longas podem ficar desatualizadas assim que todo o processo de modelagem terminar, por isso é importante mantê-lo sucinto.
Mais difícil de dar manutenção.

Código

O próprio código pode ser o modelo de domínio, o que é muito útil
Se classes e métodos estiverem bem nomeados, tudo poderá ficar claro.
Por outro lado, quem garante que o código ali dentro, a nomenclatura de variáveis e os algoritmos estarão da melhor forma e compreensíveis por todos os membros? Principalmente aqueles que não desenvolvem software.
Esse é um tópico que pode ser melhor estudado

3. Model-Driven Design

A necessidade de envolver profissionais técnicos (desenvolvedores) na modelagem

Desenvolver um modelo de negócio pode envolver analistas de negócio e especialistas de negócio, mas no momento de transcrever isso para o código, os desenvolvedores simplesmente recebem uma série de documentos e têm que implementar
Implementações podem encontrar barreiras técnicas não vistas no modelo inicial.
A alteração do modelo analítico com o modelo de código vai gerar um descompasso entre os dois caminhos, e no final o modelo analítico não vai mais ser seguido.
Futuras iterações sobre o modelo também não irão mais funcionar, já que ele não segue o código.
Por isso desenvolvedores devem participar do processo de modelagem, levantando assim visões técnicas sobre o modelo para que ele seja construído em conjunto levando todos esses aspectos em consideração.
Para ser considerada uma pessoa técnica no momento do desenvolvimento, essa pessoa deve, necessariamente, estar envolvida com a codificação em algum momento do projeto.

Sincronização entre modelo de domínio e código

Qualquer responsável por manipular código deve estar ciente que modificações no código implicam obrigatoriamente em modificações no modelo de domínio. Evitando o descompasso desses diálogos.
Sempre que essas modificações ocorrerem em qualquer lado, as partes devem se conversar, usando sempre como base a linguagem ubíqua.

Problemas de misturar código de domínio em toda a aplicação

Não isolar o domínio de outras camadas pode causar vários impactos negativos no sistema, pois qualquer alteração na lógica de domínio vai exigir uma alteração em todas as partes do sistema.
Além disso, é difícil entender o modelo de domínio se a lógica dele se espalha e enraíza por todo o código. Identificar como o domínio funciona fica bem mais complicado.

Arquitetura de camadas de uma aplicação

Camada de apresentação (interface de usuário)

Cuida de apresentar as informações para o usuário ou expor elas ao ambiente externo. Além disso, cuida da interpretação dos comandos vindas do ambiente externo. Numa aplicação MVC, cuidaria de toda a apresentação, mas numa Web API, por exemplo, apenas gerenciaria todos os comandos e retornos da API.

Camada de aplicação

Uma camada fina rensponsável pelo gerenciamento de toda a lógica entre as outras camadas. Ela não deve cuidar de regras de negócio nem de operações de persistência, por exemplo. Mas pode ser necessária como um orquestrador de comandos, por coordenar as operações entre diferentes camadas, etc.

Camada de domínio

O coração da aplicação, contendo a implementação do modelo de domínio e de todas as regras de negócio envolvidas. A persistência fica por responsabilidade da camada de infraestrutura.

Camada de infraestrutura

Camada de apoio a todas as outras, provendo comunicação entre elas, persistindo dados em arquivos, contendo libs de apoio à interface ou aplicação, etc.

Entidade

Objetos que possuem um identificador único ao longo de todo o nosso domínio, o qual não se repetirá ao longo de todo o sistema.

Objeto de valor

Objetos de valor para um sistema, mas que não precisam ter um identificador único, algo que talvez seria muito custoso de ser garantido ao longo de todo o sistema.
Pensando, por exemplo, num sistema de desenho, cada ponto de uma linha precisa ser uma entidade? Provavelmente não.
É muito adequado que objetos de valor sejam imutáveis. Ao serem considerados imutáveis, e não tendo identificador único, eles podem ser compartilhados ao longo de todo o projeto.

Serviços

Em geral, substantivos do modelo de domínio viram entidades (ou objetos de valor) e verbos viram o comportamento dessas entidades.
Porém, algumas funcionalidades de maior porte, mesmo fazendo parte do domínio como ações (verbos), vão parecer que não fazem parte de nenhum objeto. Elas podem ser passadas para um serviço.
Serviços não contém estados, mas cuidam de tratar essas operações de domínio que envolvem muitas outras entidades ou são mais complexas. Por exemplo, uma transferência entre duas contas bancárias.
Se fosse implementado diretamente nos objetos, criaria uma grande rede de dependências entre os objetos e se tornaria um código altamente acoplado.
Vale lembrar também que muitos frameworks usam o conceito de serviços para ações técnicas, o que não é um problema, mas não é o caso aqui.
É muito difícil distringuir serviços na camada de domínio — que deve ser isolada — dos serviços de infraestrutura e da camada de aplicação. O mais difícil é separar os de domínio da aplicação. Quando o serviço se refere unicamente a problemas e operações de domínio e vem para atendê-lo, ele irá nessa camada.

Módulos

Conceito

Organizam grandes partes (conjuntos) de modelos em grupos separados que são coesos e desacoplados.

Coesão comunicacional e coesão funcional

Os modelos dentro de um mesmo módulo estão juntos devido à coesão comunicacional e funcional.
Comunicacional: os modelos armazenam e conversam sobre os mesmos dados entre si;
Funcional: os modelos trabalham juntos para realizar tarefas com o mesmo propósito.

Devem expor interfaces

Para evitar o acoplamento, módulos devem expor ao mundo externo interfaces, as quais os modelos de outros módulos podem acessar. Isso evita o acoplamento que depende especificamente de implementações dos outros módulos.

Agregações

Conceito

Um grupo de objetos associados que são considerados como únicos em termos de mudança de dados. Ou seja, quem pode ser modificada é a agregação — sempre, não um dos seus objetos filhos.
A agregação forma uma fronteira e sempre tem uma raiz de agregação, que é uma entidade. Essa entidade é a única que pode se relacionar com outras entidades externas à agregação e que pode ser acessada externamente para que seja modificada.
Por exemplo, no pré-pedido, há os produtos, as variações, etc. Eles podem ser considerados uma agregação, pois sempre terei que editar o pré-pedido como um todo, e não uma variação específica.
É possível que uma raiz de agregação passe seus objetos internos a outros interessados externos na forma de cópias de objetos de valor ou entidades, desde que esses não guardem ou modifiquem essas referências. Por isso o fato de enviar cópias.

Vantagens

Fica muito mais fácil de gerenciar regras nos relacionamentos, pois quem gerencia todas elas é sempre a raiz de agregação, nunca outros envolvidos.
Se a raiz de agregação for excluída, é simples, todas as outras raízes são também excluídas.

Factories

A criação de objetos complexos, principalmente das raízes de agregação, não deveria ser uma responsabilidade dos objetos em si em seus construtores. Isso faz com que objetos, usados por exemplo na camada de aplicação, tragam junto toda essa lógica, ferindo vários conceitos importantes do desenvolvimento de software.
O padrão é usado principalmente para agregar toda a lógica de criação em uma fábrica separada, que garante o respeito a todas as relações entre objetos.
Possíveis design patterns sugeridos: Factory method e Abstract factory, a serem estudados melhor posteriormente.

Repositórios

É uma boa opção manter entidades como referências?

Não é uma boa opção, pois isso gera acoplamento de código, já que todo mundo que precisar utilizar de uma raiz de agregação ao longo do código vai precisar manter essa referência guardada.
Além disso, em algum momento a aplicação pode cair, e vai ser necessário buscar essas referências novamente, em um banco de dados relacional ou em qualquer outro sistema de persistência.

Por que criar uma classe específica para o repositório?

Se o banco de dados puder ser acessado diretamente no domínio, toda a lógica de infraestrutura vai ficar dispersa ali dentro. O domínio vai ficar ciente de lógicas de acesso ao banco de dados e, por exemplo, se precisar mudar o SGBD que está sendo acessado, vários códigos espalhados terão que ser alterados.
Ainda, se o código do cliente (camada de aplicação) ficar responsável pelas lógicas de acesso, ele vai poder montar queries para buscar exatamente os dados que precisa, e a camada de domínio se tornará um mero container de dados.

Função do repositório

Fica responsável por toda a lógica necessária para obter referências de objetos. Por exemplo, se um objeto já está na memória e foi solicitado novamente, o repositório pode evitar uma consulta no banco e apenas retornar tal referência.
Os repositórios devem ser focados em raízes de agregação que precisam de acesso direto.
A implementação do repositório será puramente associada à infraestrutura, mas vale destacar que a sua interface será puramente associada com o modelo de domínio.

4. Refactoring Toward Deeper Insight

Trazendo conceitos chave do modelo à tona

Formas de trazer conceitos chave mais complexos à tona na modelagem:

Constraints

Geralmente expresso na forma de um método dentro de uma entidade, ele é o limitador de alguns fatores existentes na própria entidade. Se o limite for extrapolado a entidade fica inválida, por exemplo lançando uma exceção. Um exemplo é uma prateleira, que não pode ter mais livros do que a sua capacidade.

Process

Processos são as partes relacionadas ao domínio e que devem ficar nele representadas como um serviço. São os algoritmos mais complexos, relacionamentos de entidades, etc.

Specification

Especificação é uma forma de validar determinado objeto, geralmente com respostas de “sim” ou “não”. Em geral, se são especificações simples, elas podem ficar dentro do próprio objeto para testar se ele atende a determinados critérios. Não quer dizer que o objeto é válido ou não, mas sim para que isso seja usado para outros aspectos. Por exemplo, um cliente, verificando se ele é elegível para uma linha de crédito ou não.
As especificações mais complexas podem ficar em objetos separados. Pois, por exemplo, verificar se um cliente pode fazer parte de uma linha de crédito requer muitas outras informações importantes que envolvem outros objetos, e colocar tudo isso como dependência do objeto Cliente seria gerar muitas conexões e acoplamentos.

5. Preserving Model Integrity

Bounded Context

Qual deve ser o tamanho de um modelo?

Um modelo deve ser pequeno o suficiente para que um time possa ser responsável por ele.

Como separar modelos em contextos?

É importante separar os modelos para que cada time fique responsável por eles. Para isso, é muito importante direcionar algum esforço à etapa de definição de limites de cada contexto. Isso inclui:
Definir os limites de partes da aplicação (serviços).
Definir os limites de modelos relacionais de bancos de dados, sendo quem cada um ficará responsável por gerenciar.
Definir os limites do próprio modelo e de quando cada um pode ou não armazenar diferentes informações sobre determinado modelo.

Qual a diferença entre Bounded Context e Module?

Um módulo agrupa e organiza diferentes partes de um domínio, enquanto o Bounded Context pode agregar vários módulos dentro de si de acordo com os inúmeros boundaries definidos para a organização.

Integração contínua (Continuous Integration) de Bounded Context

Conceito

É muito importante manter o modelo sempre atualizado, e a equipe precisa estar toda muito ciente e focada 100% na integridade do modelo, para assim terem noção de que qualquer alteração precisa encaixar corretamente no modelo para todos, sem alterações arbitrárias a todo o momento.

Boas práticas e processos de integração contínua

A integração contínua é um processo que deve existir quando se trata de Domain Driven Design para que o modelo esteja sempre o mais próximo possível da versão mais atualizada. Ou seja, uma nova concepção do modelo não fica muito tempo parada numa branch, por exemplo. Alguns processos que podem ajudar nisso:
Aderência do time aos testes automatizados. Inclusive usando a prática do TDD, que garante que nada no código sofrerá merge sem que exista um teste para isso e todo o modelo permaneça íntegro.
Merges para a branch principal com a maior frequência possível. Assim, os devs garantem sempre ter a versão mais atualizada possível e não ocorra sobreposição no trabalho de dois desenvolvedores. A cadência de merges diários é uma ótima opção.
Builds automatizados para que os merges diários + os testes sempre rodem automaticamente.
A integração contínua se aplica apenas a um Bounded Context e nunca às relações entre Bounded Contexts.

O que é um Context Map

É um mapa, geralmente um diagrama ou na forma de texto, que apresenta todos os Bounded Contexts e suas relações. É muito importante que todos os desenvolvedores tenham conhecimento dele e o compartilhem.

Formas/padrões de interação entre diferentes contextos

Shared Kernel

Em uma organização que não tenha políticas tão claras quanto o desenvolvimento e modelos de domínio muito complexos, é muito custoso manter a integração contínua, seja porque times são muito grandes ou não são capazes de manter a comunicação sobre um modelo único. Por isso, é uma boa ideia criar Bounded Contexts para cada equipe e separá-las.
Porém, as organizações (e equipes criadas) terão partes do modelo similares e vão juntas “correr” para um mesmo objetivo, mas em algum momento entrarão em descompasso e cada um dos modelos de domínio não irá mais encaixar. A partir daí, o esforço com a integração contínua, testes e tradução do modelo entre as aplicações pode crescer em níveis desenfreados.
Por isso, vem o shared kernel, um conceito onde as equipes concordam em compartilhar um pedaço do modelo de domínio. Qualquer alteração nesse pedaço deve ser refletida para ambas as equipes, assim como é o processo de integração contínua para um Bounded Context. Além disso, todas as alterações devem ser feitas com o consenso de todos.
Ainda, o shared kernel deve seguir todos os passos do Bounded Context:
contar com testes automatizados.
builds frequentes, mas não tão frequentes quanto os internos do Bounded Context. Talvez semanalmente seja uma cadência adequada.
ao realizar os builds, o set de testes automatizados deve ser realizado para ambas as equipes.
O propósito do Shared Kernel é que os Bounded Contexts continuem separados, mas a duplicação seja reduzida trabalhando com um código compartilhado.

Customer-Supplier

Há muitos casos de organizações onde há uma relação pai-filho nas dependências de domínio dos projetos. No exemplo citado no livro, de uma aplicação de e-commerce e uma aplicação de relatórios, a aplicação principal (e-commerce) não tem preocupação com nada do domínio da parte de relatórios; por outro lado, toda a parte de relatórios depende do domínio e modelagem da parte de e-commerce para agrupamento, pivoteamento e apresentação dos dados.
Porém, haverá informações que não têm significado nenhum para a aplicação de e-commerce que significam muito para a aplicação de relatórios. Assim, há uma relação de cliente-fornecedor (customer-supplier).
Esse padrão funciona bem quando os dois times estão atuando sob a mesma gerência, facilitando as relações. Isso porque o esquema do banco de dados, por exemplo, pode precisar ser mudado por ambas as aplicações, e é difícil imaginar que ele não vá mudar ao longo do desenvolvimento.
O time de relatórios atua como o cliente: apresenta as informações pedidas em reuniões regulares ou quando necessário. O time do e-commerce, com tais informações, apresenta os planos e o prazo para tal.
Apesar de que mudanças no sistema podem ser realizadas à vontade pela equipe de e-commerce, a interface entre as duas aplicações deve sempre permanecer a mesma, acompanhada por uma suíte de testes de ambas as partes. Essa interface só pode ser alterada com acordo de ambas as partes.

Conformist

Há alguns casos, principalmente em times que não trabalham sob a mesma gerência, onde o time fornecedor não conseguirá atender o time cliente por quaisquer motivos, ou não será do interesse deles, por falta de organização, pressão, ou porque o time fornecedor tem suas próprias demandas e depende unicamente do altruísmo para atender o time cliente. A relação customer-supplier funciona apenas quando ambos os times estão interessados na relação.
Assim, o time cliente tem como opção se tornar um conformista. Ele adotará integralmente o modelo do time fornecedor e se conformar inteiramente a ele, não podendo realizar modificações.
O time cliente terá que implementar uma camada anticorrupção, seja para se proteger das mudanças no modelo ou para se adaptar a modelos pobres.

Camada anticorrupção

Comentado no tópico acima, alguns casos podem levar à necessidade de uma camada anticorrupção:
Aplicações legadas construídas sem técnicas de modelagem de domínio podem ter um modelo confuso, difícil de entender e até inútil;
Aplicações novas têm de se proteger de mudanças nas aplicações fornecedoras/legadas.
→ Aplicações novas geralmente acessam aplicações legadas via banco de dados.
Aplicações clientes e fornecedoras podem se comunicar de várias maneiras, uma delas é via rede, mas a forma mais comum é que sistemas legados sejam acessados diretamente pelo banco de dados. Por isso, é importante destacar que o banco de dados é a forma de armazenar os dados, mas não necessariamente expressa o que o modelo quer dizer. Assim, aqui entra novamente a camada anticorrupção.
A camada anticorrupção funciona como uma interface entre o modelo antigo e o novo. Ela conversa com o modelo antigo conforme as suas regras de domínio e impede que o sistema novo invada ou altere a forma que o sistema legado deve se comportar.
A implementação funciona geralmente por meio de um serviço, o qual funcionará como o design pattern facade. Ele provê uma interface para que o modelo de domínio novo acesse e manipule informações do modelo de domínio antigo, isolando-os integralmente.
